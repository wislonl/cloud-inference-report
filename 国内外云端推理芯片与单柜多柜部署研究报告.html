Cloud Inference Chips · Rack Strategy · 2026-04-22

把芯片选择、系统互联、机柜热密度，放到同一张决策图里。

这不是单纯的“谁的 TOPS 更高”报告，而是一个面向部署落地的观察框架：大模型推理进入显存与系统工程时代，小模型服务则继续由功耗、成本与运维复杂度主导。这个单页版本面向现场演示、移动端阅读，以及按需改字后导出 PDF。

核心判断 02 30kW/柜后，机房已经进入方案本体

散热、配电、故障域、网络层级不再是外部条件，而是推理成本和可扩展性的组成部分。

第一优先级 显存容量与带宽

决定 70B 以上模型能否装得下、跑得稳、上下文拉得长。

第二优先级 片间互联

决定你是做单节点副本，还是被迫走跨节点分片。

第三优先级 软件栈成熟度

推理引擎、量化、continuous batching 会显著改变 tokens/美元。

第四优先级 机柜热密度

30kW/柜以上优先直液冷，60kW/柜以上不宜按传统风冷思路扩容。

执行摘要

这份网页把原文拆成“结论先行 + 决策路径 + 平台卡片 + 典型案例”的阅读顺序，更适合现场演示、手机浏览和后续再编辑。

2026 年的云端推理主战场，不再是单一的 TOPS/FLOPS，而是显存容量与带宽、片间互联、推理软件栈、机柜热密度四个维度的耦合优化。
70B–405B 模型更受益于高显存、高带宽与强片间互联，尤其在长上下文、长 KV cache 与多并发服务中。
7B–13B、CV、BERT 类任务通常在 L4/L40S、Inferentia2、Atlas 300I、BM1684X 一类平台上更容易做出性价比。
国产平台在数据主权、供应链韧性和政企私有化上有天然优势，但公开、可复核 benchmark 透明度仍弱于国际 GPU/TPU/云 ASIC 阵营。

建议先回答三个问题

先问网络：你的价值在单机箱互联，还是跨机架扩展？
先问热密度：设计功率接近 30kW/柜后，传统风冷的可运维性会迅速下降。
先问故障域：做单节点大副本，还是多节点分片？这两条路对网络、配电和运维策略要求完全不同。

选型矩阵

先按负载类型和组织约束分层，再去看单芯片算力。否则容易在采购时被“峰值数字”误导。

Small Models

小模型、BERT、ResNet、检索推荐

目标通常是低时延、低功耗、低单次请求成本。

  • 更关键的指标：单请求延迟、批处理开销、每百万请求成本。
  • 更合适的平台：L4/L40S、Inferentia2、Atlas 300I Duo、BM1684X。
  • 常见误区：一开始就上 8×H200/MI325X。
7B–70B

中大模型在线服务

从这个区间开始，显存容量、HBM 带宽和 KV cache 管理已经变成一线问题。

  • 更关键的指标：显存容量、HBM 带宽、continuous batching、量化路径。
  • 更合适的平台：H100 NVL/H200、MI300X、Gaudi3、TPU v5e/v6e、Ascend 私有化方案。
  • 常见误区：只看峰值 FLOPS。
70B–405B

长上下文与高并发 LLM

系统架构和网络设计开始超过单卡压测的重要性。

  • 更关键的指标：片间互联、张量并行、TTFT、尾延迟与故障域。
  • 更合适的平台：H200、MI325X、Trillium、Trainium2、B200、Ascend 超节点。
  • 常见误区：用单副本思维扩容。
Compliance

主权、合规、连续可得性

当数据不出域、行业监管强或供应链波动大时，平台选择逻辑会发生变化。

  • 优先纳入 PoC：昇腾、寒武纪、算能。
  • 必须增加的维度：框架迁移成本、推理精度回归、行业链路可得性、本地服务能力。
  • 更稳妥的策略：核心大模型与轻量感知任务分平台部署。

任务类型 模型规模 推荐平台 原因 不推荐方案 关键风险
通用 LLM 在线问答 7B-13B 单机 4-8 卡国产推理卡或国际主流 GPU 模型可单节点落地，建设成本可控，适合知识库问答与内部 Copilot。 直接上超大集群或高规格液冷方案 软件栈适配不足时，吞吐达标但时延不稳定。
企业内部知识助手 14B-32B 单机 8 卡 GPU 服务器 兼顾精度、响应速度与后续扩容空间，适合中型企业私有化。 仅用 CPU 集群硬顶大模型 高峰并发时 TTFT 和 P99 易劣化。
代码生成与复杂推理 34B-70B 8 卡高带宽互联 GPU 节点 显存带宽、框架兼容和张量并行要求高，成熟生态能显著降低调优周期。 低互联带宽的多机松耦合集群 多卡通信开销过大，实际吞吐可能低于预期。
多租户推理云服务 7B-70B 混部 国际主流 GPU 平台或生态成熟的国产平台 调度工具链和监控体系更完善，利于多租户运营。 仅按单模型满配采购 资源碎片化严重，利用率偏低。
大模型蒸馏后的行业模型 7B-14B 国产推理卡优先 成本、供货与本地化服务更有优势，适合政企定制部署。 沿用训练型高端 GPU 采购成本高，单位 token 成本不划算。
超大模型研究验证 70B-405B 多节点高端 GPU 集群 需要成熟的并行框架、RDMA 网络和稳定驱动。 单柜内强行堆叠完成全量部署 机柜功耗、散热和通信瓶颈难以收敛。
边缘侧或分支机构推理 3B-7B 小型国产加速卡或低功耗 GPU 更关注功耗、体积和易维护性，适合轻量问答与分类任务。 数据中心级高功耗平台下沉 成本和运维复杂度明显过高。

单柜 / 多柜成本拆分

把 CAPEX 与 OPEX 拆开看，才能解释为什么“看似只是多买几台服务器”，最后却演变成网络、散热、配电和运维体系的重投入。

成本项 单柜部署口径 多柜部署口径 成本变化趋势 说明
服务器1-4 台高密度 GPU 服务器随节点数增加线性上升显著上升CAPEX 最大项，决定基础算力与显存容量。
交换机柜内 ToR 或较低规模高速交换需增加 Spine/Leaf 或更高规格高速交换快速上升多柜部署后，网络成本占比明显提升。
存储本地 NVMe 或小规模共享存储集中式高性能存储或分布式存储中度上升模型权重、日志与检查点统一管理成本增加。
液冷 / 风冷中低功率场景可维持高规格风冷高功率密度场景通常需要液冷或行级制冷阶跃上升功率密度超过机房承载能力时，冷却成本突增。
配电单柜内配电改造相对简单需综合评估 UPS、PDU、母线及冗余设计中高上升多柜部署对电力冗余和持续供电能力要求更高。
机柜单柜成本可控机柜数量、布线与占地同步增加线性上升常被低估，但会持续推高基础设施投入。
电费功耗总量相对可控节点数上升后电费成为核心 OPEX持续上升大模型常态化推理时，电费会长期侵蚀单位 token 成本。
运维以单点维护、人工值守为主需引入监控、调度、容量管理和故障演练体系非线性上升多柜后运维复杂度往往高于硬件数量的线性增长。
总体来看，单柜部署适合 PoC、试运营和中小规模私有化场景，优势是决策快、建设周期短、CAPEX 可控；多柜部署适合高并发、多模型混部及超大模型场景，扩展性强，但网络、制冷、配电与运维体系会显著抬升总体拥有成本。

国产 vs 国际平台 PoC 清单

PoC 不应只验证“模型能不能跑通”，更应验证“上线后能不能稳定交付”。统一指标口径，比单看厂商 benchmark 更有管理价值。

评估维度 验证要点 国产平台关注点 国际平台关注点 建议验收口径
框架兼容PyTorch、vLLM、TensorRT-LLM、主流 serving 框架是否稳定重点确认适配版本、算子覆盖率与社区插件完整性生态更成熟，但仍需确认特定模型支持情况目标模型与目标框架完成全链路跑通
量化精度INT8、FP8、BF16 等精度下效果损失是否可控确认量化工具链与校准流程成熟度方案更丰富，但需关注许可和版本绑定核心业务任务精度损失在可接受范围内
吞吐单位时间 token 输出能力是否满足目标并发重点看实测而非理论峰值排查不同 batch 策略下的性能波动统一 prompt、batch、上下文长度后测得吞吐
TTFT首 token 返回时延是否满足交互体验驱动与调度优化不足时 TTFT 易偏高成熟生态通常优势更明显P50、P95 TTFT 同时达标
P99高峰尾时延是否可控多租户和高并发下更需重点验证确认持续压测下是否出现抖动P99 不超过业务 SLA 上限
驱动成熟度长时间运行是否稳定，升级是否可控看驱动、固件、编译链一致性看版本兼容矩阵和升级风险连续稳定运行 72 小时以上无严重故障
供货周期设备是否能按项目节奏交付确认批量交付能力和区域供应稳定性关注高端卡配额与地缘供应风险形成明确交期承诺与替代方案
售后能力问题响应、驻场支持、备件体系是否完善本地化支持通常是优势点确认国内响应链路和服务时效约定故障响应时间、备件与升级支持条款
PoC 阶段建议坚持“统一模型、统一数据集、统一并发口径、统一统计指标”的原则，避免仅依据厂商单点 benchmark 做结论。最终应关注平台是否能在目标预算、目标交期与目标 SLA 下稳定交付。

70B / 405B 部署边界

70B 与 405B 不只是参数差异，更对应不同的节点组织方式、网络要求与制冷方式。这里把部署边界直接写清楚，便于前期排除不现实方案。

模型规模 典型精度 建议部署方式 节点数量建议 网络要求 散热要求 适用场景 主要边界条件
70BINT4/INT8单节点优先，必要时双节点扩副本1 节点起步RoCE 非刚需，万兆可起步高端风冷可覆盖企业知识问答、代码助手前提是并发不高、上下文长度受控
70BFP8/BF16双节点或多节点更稳妥2-4 节点建议 100G 以上互联高密部署建议液冷高质量生成、复杂推理显存占用和功耗显著抬升，单节点空间紧张

边界判断可以简化成一句话：70B 更适合先做单柜闭环，405B 更适合按多柜体系设计。不要把演示级配置误判为生产级方案。

典型采购建议

采购不应只按“芯片先进程度”排序，而应把预算边界、交付周期、业务连续性和运维能力一起纳入判断。

创业团队

推荐起步配置：1 台推理服务器 + 基础存储 + 千兆/万兆网络，优先国际成熟生态或国产成熟 SKU。

  • 采购重点：先把模型跑通、把 API 服务稳定。
  • 不建议踩坑：一开始就采购多柜和复杂网络。
  • 适用阶段：0 到 1 产品验证。

中型企业私有化

推荐起步配置：2-4 台服务器 + 基础共享存储 + 万兆/25G 网络，优先兼容主流框架且售后明确的平台。

  • 采购重点：交付周期、运维简化、总拥有成本。
  • 不建议踩坑：只看芯片峰值参数，忽略系统集成。
  • 适用阶段：部门级到公司级内部落地。

政企 / 运营商

推荐起步配置：单柜起步，预留多柜扩展能力，配套交换与供配电方案，优先供货稳定、国产化路径清晰的平台。

  • 采购重点：合规、供应保障、售后 SLA、生态适配。
  • 不建议踩坑：用实验室配置直接做正式采购。
  • 适用阶段：试点到规模化部署。

科研 / HPC

推荐起步配置：多节点集群 + 高速互联 + 并行文件系统，优先高带宽互联与大模型并行能力强的平台。

  • 采购重点：混部能力、作业调度、精度支持。
  • 不建议踩坑：只按单卡指标判断整体效率。
  • 适用阶段：科研训练与超大模型推理。
通用建议是分两阶段采购：先做 PoC 小批量验证，验证后再做规模采购。合同中应写清供货周期、驱动版本、售后边界，并尽量保留替代供应与回退路线。

来源可信度分层

正式报告中的数据，必须区分“可直接引用的事实”和“用于工程判断的参考”。这样答辩时更稳，也更容易解释结论边界。

一级到三级：可作为硬依据

  • 官方规格：官网 datasheet、白皮书，适合确认静态能力边界。
  • 云厂商价格：用于租赁成本和初步 TCO 估算，需注明时间点与地区。
  • MLPerf / 公开 benchmark：适合做横向参考，但要核对模型、批大小和精度设置。

四级到六级：用于补充与风险提示

  • 厂商材料：可了解方向，但不能单独作为采购依据。
  • 第三方实测：对工程判断有价值，但必须交叉验证。
  • 生态推断：适合形成趋势判断，必须明确标注为“推断”。

一级：官方规格厂商正式发布的产品规格、接口、功耗说明官网 datasheet、白皮书芯片参数、服务器规格、接口数量可直接引用，但不代表实际业务性能
二级：云厂商价格公开售卖或报价体系中的实例价格与计费信息公有云官网、官方报价单租赁成本、初步 TCO 估算需注明时间点与地区差异
三级：公开 benchmark具备公开方法论和可复核条件的测试结果MLPerf、公开评测报告吞吐、延迟、单位成本比较需核对测试模型、batch 与精度设置
四级：厂商材料厂商方案书、宣传 PPT、联合发布材料市场材料、路演材料场景案例、生态合作、能力宣称仅作为选型参考，不单独作结论
五级：第三方实测集成商、媒体、社区或客户侧实测反馈技术博客、社区评测驱动成熟度、安装复杂度、真实稳定性需交叉验证，不宜孤证定论
六级：生态推断根据框架支持、社区活跃度、供货信息做综合判断兼容清单、社区 issue、渠道信息供货风险、生态成熟度、未来可维护性必须明确标注为“推断”

≤ 1 柜 7B–70B 低时延私有部署

单机柜低时延大模型服务柜

参考配置为 1 台 8×Gaudi 3 节点 + 双 200GbE ToR + NVMe 缓存/日志节点，适合 RAG、长上下文 API 和私有服务入口。

  • IT 功耗约 9.2kW，PUE 1.35，月电费约 7,154 元。
  • 若业务以 70B 为主且强调极低 TTFT，这种柜型非常合适。
  • 主要风险：RoCE 参数、NUMA 亲和、容器镜像与推理框架版本一致性。
4 柜 70B–405B 多租与分片

四机柜中型 H200 推理集群

参考配置为 4 台 DGX H200 + 2 台 NDR400 IB 交换机 + 管理/存储交换机与并行文件或对象存储节点。

  • IT 功耗约 46.8kW，PUE 1.25，月电费约 33,696 元。
  • 建议拆成两个故障域，并把 prefill 与 decode 分开调度。
  • 主要风险：跨机架 TTFT、KV cache 拆分、IB/RDMA 抖动、升级窗口管理。
12 柜示例 > 405B / MoE 推理工厂

大规模多柜 GB200 机架级推理工厂

示例按 12 个 GB200 NVL72 计算柜估算，适合 trillion 参数级推理、实时多模态与极大并发长上下文。

  • 主要风险：机房土建、液冷、电力先行，以及单柜故障影响面过大。

封面战略总览

把芯片选择、系统互联、机柜热密度，放到同一张决策图里。

大模型推理进入显存与系统工程时代，小模型服务则继续由功耗、成本与运维复杂度主导。芯片选择、软件栈、机柜热密度与网络拓扑，已经必须作为同一件事同时评估。

核心判断 01

  • 70B–405B 更依赖高显存、高带宽与强片间互联。
  • 长上下文、长 KV cache、高并发会放大片间通信成本。

核心判断 02

  • 30kW/柜之后，机房条件本身就是方案的一部分。
  • 散热、配电、故障域和网络结构会直接影响 tokens/美元。
第二页选型方法

先按负载类型分层，再看单芯片参数。

小模型与感知任务

  • 先看时延、成本和运维复杂度。
  • L4/L40S、Inferentia2、Atlas 300I、BM1684X 更常见。

中大模型在线服务

  • 7B–70B 开始，显存容量与量化路径变成一线问题。
  • 70B–405B 开始，系统架构往往比单卡压测更重要。
第三页平台图谱

国际阵营更透明，国产阵营更可控。

国际平台

  • GPU、云 ASIC、超节点路线都已经成熟。
  • 优点集中在公开 benchmark、推理生态与价格透明度。

国产平台

  • 适合数据主权、行业交付与私有化场景。
  • PoC 前要重点验证框架兼容与精度回归。
第四页部署案例

不同规模的最优策略，并不相同。

单机柜

  • 适合 7B–70B 低时延私有服务。
  • 主要风险来自网络参数、NUMA 亲和和镜像一致性。

多机柜

  • 70B–405B 后更要重视故障域、prefill/decode 分离和跨机架通信。
  • 大规模工厂化方案必须把液冷、电力、网络与平台软件放进同一项目组。
第五页结论

采购前先固定场景，再固定指标，再固定扩容路径。

如果只记住一句话：别用“大模型推理工厂”的方法做小规模部署，也别用“小模型轻量化”的思路去判断 405B 级服务集群。最优解取决于任务、网络、热密度和供应链边界是否匹配。

适合小规模

  • BERT、推荐、OCR、视频理解、7B–13B 优先看低功耗高性价比平台。
  • PoC 阶段必须固定输入长度、并发和量化精度。

适合中大规模

  • 把网络当一等公民设计，至少分离计算、管理、存储三张网。
  • 数据不出域或监管强时，应把国产路线放进同框 PoC。
