第一优先级 显存容量与带宽

第二优先级 片间互联

第三优先级 软件栈成熟度

第四优先级 机柜热密度

30kW/柜以上优先直液冷，60kW/柜以上不宜按传统风冷思路扩容。

资料库总览

2026 年的云端推理主战场，不再是单一的 TOPS/FLOPS，而是显存容量与带宽、片间互联、推理软件栈、机柜热密度四个维度的耦合优化。
70B–405B 模型更受益于高显存、高带宽与强片间互联，尤其在长上下文、长 KV cache 与多并发服务中。
7B–13B、CV、BERT 类任务通常在 L4/L40S、Inferentia2、Atlas 300I、BM1684X 一类平台上更容易做出性价比。
国产平台在数据主权、供应链韧性和政企私有化上有天然优势，但公开、可复核 benchmark 透明度仍弱于国际 GPU/TPU/云 ASIC 阵营。

建议先回答三个问题

先问网络：你的价值在单机箱互联，还是跨机架扩展？
先问热密度：设计功率接近 30kW/柜后，传统风冷的可运维性会迅速下降。
先问故障域：做单节点大副本，还是多节点分片？这两条路对网络、配电和运维策略要求完全不同。

Benchmark / 来源图

把这份资料库后续最该引用的 4 类证据拆开。这样你以后补新数据时，知道应该先找哪类来源，也知道每类来源能回答什么问题。

Level 1 官方规格页

最适合回答静态能力边界：显存、带宽、互联、功耗、形态。

  • NVIDIA / AMD / Intel 产品页
  • Google TPU 文档、AWS Neuron 架构文档
  • 昇腾 / 寒武纪 / SOPHGO 官方规格页
Level 2 MLPerf v6.0

最适合回答可公开横比的标准 benchmark，尤其是国际主线平台。

  • 只对统一模型、统一场景、统一规则有效
  • 适合解释公开 benchmark 透明度
  • 不适合替代真实业务 PoC
Level 3 云实例价 / 平台价格页

最适合回答 tokens / 美元、租赁成本、上云试错成本。

  • AWS / Google / CoreWeave / 云厂商实例页
  • 要注明时间点、区域和实例规格
  • 适合辅助判断，不是芯片静态能力
Level 4 官方性能页 / 白皮书

最适合回答特定模型或特定负载下的方向性优势，但不能和 MLPerf 混成一个总表。

  • NVIDIA H200/H100 性能页
  • AMD Instinct 比较页
  • Intel Gaudi 白皮书与性能分析
后面你继续补内容时，最稳的顺序是：先找官方规格，再找 MLPerf / 官方 benchmark，再找云价，再找厂商性能页。这样这份资料库会越来越完整，但不会越来越乱。

选型矩阵

先按负载类型和组织约束分层，再去看单芯片算力。否则容易在采购时被“峰值数字”误导。

Small Models

小模型、BERT、ResNet、检索推荐

目标通常是低时延、低功耗、低单次请求成本。

  • 更关键的指标：单请求延迟、批处理开销、每百万请求成本。
  • 更合适的平台：L4/L40S、Inferentia2、Atlas 300I Duo、BM1684X。
  • 常见误区：一开始就上 8×H200/MI325X。
7B–70B

中大模型在线服务

从这个区间开始，显存容量、HBM 带宽和 KV cache 管理已经变成一线问题。

  • 更关键的指标：显存容量、HBM 带宽、continuous batching、量化路径。
  • 更合适的平台：H100 NVL/H200、MI300X、Gaudi3、TPU v5e/v6e、Ascend 私有化方案。
  • 常见误区：只看峰值 FLOPS。
70B–405B

长上下文与高并发 LLM

系统架构和网络设计开始超过单卡压测的重要性。

  • 更关键的指标：片间互联、张量并行、TTFT、尾延迟与故障域。
  • 更合适的平台：H200、MI325X、Trillium、Trainium2、B200、Ascend 超节点。
  • 常见误区：用单副本思维扩容。
Compliance

主权、合规、连续可得性

当数据不出域、行业监管强或供应链波动大时，平台选择逻辑会发生变化。

  • 优先纳入 PoC：昇腾、寒武纪、算能。
  • 必须增加的维度：框架迁移成本、推理精度回归、行业链路可得性、本地服务能力。
  • 更稳妥的策略：核心大模型与轻量感知任务分平台部署。

总体来看，单柜部署适合 PoC、试运营和中小规模私有化场景，优势是决策快、建设周期短、CAPEX 可控；多柜部署适合高并发、多模型混部及超大模型场景，扩展性强，但网络、制冷、配电与运维体系会显著抬升总体拥有成本。

典型采购建议

采购不应只按“芯片先进程度”排序，而应把预算边界、交付周期、业务连续性和运维能力一起纳入判断。

创业团队

推荐起步配置：1 台推理服务器 + 基础存储 + 千兆/万兆网络，优先国际成熟生态或国产成熟 SKU。

  • 采购重点：先把模型跑通、把 API 服务稳定。
  • 不建议踩坑：一开始就采购多柜和复杂网络。
  • 适用阶段：0 到 1 产品验证。

中型企业私有化

推荐起步配置：2-4 台服务器 + 基础共享存储 + 万兆/25G 网络，优先兼容主流框架且售后明确的平台。

  • 采购重点：交付周期、运维简化、总拥有成本。
  • 不建议踩坑：只看芯片峰值参数，忽略系统集成。
  • 适用阶段：部门级到公司级内部落地。

政企 / 运营商

推荐起步配置：单柜起步，预留多柜扩展能力，配套交换与供配电方案，优先供货稳定、国产化路径清晰的平台。

  • 采购重点：合规、供应保障、售后 SLA、生态适配。
  • 不建议踩坑：用实验室配置直接做正式采购。
  • 适用阶段：试点到规模化部署。

科研 / HPC

推荐起步配置：多节点集群 + 高速互联 + 并行文件系统，优先高带宽互联与大模型并行能力强的平台。

  • 采购重点：混部能力、作业调度、精度支持。
  • 不建议踩坑：只按单卡指标判断整体效率。
  • 适用阶段：科研训练与超大模型推理。
通用建议是分两阶段采购：先做 PoC 小批量验证，验证后再做规模采购。合同中应写清供货周期、驱动版本、售后边界，并尽量保留替代供应与回退路线。

来源可信度分层

正式报告中的数据，必须区分“可直接引用的事实”和“用于工程判断的参考”。这样答辩时更稳，也更容易解释结论边界。

一级到三级：可作为硬依据

  • 官方规格：官网 datasheet、白皮书，适合确认静态能力边界。
  • 云厂商价格：用于租赁成本和初步 TCO 估算，需注明时间点与地区。
  • MLPerf / 公开 benchmark：适合做横向参考，但要核对模型、批大小和精度设置。

  • 厂商材料：可了解方向，但不能单独作为采购依据。
  • 第三方实测：对工程判断有价值，但必须交叉验证。
  • 生态推断：适合形成趋势判断，必须明确标注为“推断”。

封面战略总览

把芯片选择、系统互联、机柜热密度，放到同一张决策图里。

大模型推理进入显存与系统工程时代，小模型服务则继续由功耗、成本与运维复杂度主导。芯片选择、软件栈、机柜热密度与网络拓扑，已经必须作为同一件事同时评估。

核心判断 01

  • 70B–405B 更依赖高显存、高带宽与强片间互联。
  • 长上下文、长 KV cache、高并发会放大片间通信成本。

核心判断 02

  • 30kW/柜之后，机房条件本身就是方案的一部分。
  • 散热、配电、故障域和网络结构会直接影响 tokens/美元。
第二页资料库架构

这页已经从报告，升级成资料库入口。

双入口

  • 专题库负责横向比较：性能、成本、部署、采购、来源。
  • 品牌库负责纵向追踪：型号、互联、生态、更新记录。

长期维护

  • 新增信息优先补专题或品牌，不再往首页堆长文。
  • 旧资料更新统一写入更新区，明确日期与口径变化。
第三页选型方法

先按负载类型分层，再看单芯片参数。

小模型与感知任务

  • 先看时延、成本和运维复杂度。
  • L4/L40S、Inferentia2、Atlas 300I、BM1684X 更常见。

中大模型在线服务

  • 7B–70B 开始，显存容量与量化路径变成一线问题。
  • 70B–405B 开始，系统架构往往比单卡压测更重要。
第四页品牌库

国际阵营更透明，国产阵营更可控。

国际平台

  • GPU、云 ASIC、超节点路线都已经成熟。
  • 优点集中在公开 benchmark、推理生态与价格透明度。

国产平台

  • 适合数据主权、行业交付与私有化场景。
  • PoC 前要重点验证框架兼容与精度回归。
第五页部署案例

不同规模的最优策略，并不相同。

单机柜

  • 适合 7B–70B 低时延私有服务。
  • 主要风险来自网络参数、NUMA 亲和和镜像一致性。

多机柜

  • 70B–405B 后更要重视故障域、prefill/decode 分离和跨机架通信。
  • 大规模工厂化方案必须把液冷、电力、网络与平台软件放进同一项目组。
第六页更新治理

采购前先固定场景，再固定指标，再固定扩容路径。

如果只记住一句话：别用“大模型推理工厂”的方法做小规模部署，也别用“小模型轻量化”的思路去判断 405B 级服务集群。最优解取决于任务、网络、热密度和供应链边界是否匹配；资料库更新也必须按统一口径进入。

适合小规模

  • BERT、推荐、OCR、视频理解、7B–13B 优先看低功耗高性价比平台。
  • PoC 阶段必须固定输入长度、并发和量化精度。

适合持续更新

  • 把网络当一等公民设计，至少分离计算、管理、存储三张网。
  • 新品牌、新型号、新 benchmark 统一先归入专题库或品牌库，再升级到首页结论。
